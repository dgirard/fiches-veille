1 Architecture globale

L'Agent view s'appuie sur un processus superviseur par utilisateur qui héberge toutes les sessions en arrière-plan. Il démarre automatiquement au premier dispatch et reste indépendant du terminal.

2 Cycle de vie d'une session

Une session passe par plusieurs états. Après ~1h d'inactivité, le superviseur arrête le process pour libérer des ressources — le transcript et l'état restent sur disque et un nouveau process est relancé au prochain accès.

stateDiagram-v2 [*] --> Working: dispatch Working --> NeedsInput: permission ou question NeedsInput --> Working: reply Working --> Idle: pause Idle --> Working: nouvelle instruction Working --> Completed: result Working --> Failed: erreur Working --> Stopped: Ctrl+X ou claude stop Completed --> ProcessExit: 1h sans attach Failed --> ProcessExit: 1h sans attach Stopped --> ProcessExit: immediat ProcessExit --> Working: attach peek reply (respawn) Stopped --> [*]: Ctrl+X x2 (delete)

5 Trois manières de dispatcher

A · Depuis Agent view

Tape un prompt dans l'input du bas + Enter.

  • @<repo> — cible un repo voisin
  • @<agent> — utilise un subagent
  • /<skill> — lance une skill
  • #<PR> — session existante sur ce PR
  • Shift+Enter — dispatch + attach immédiat

B · Depuis une session interactive

  • /bg [prompt] — backgroundise la session
  • ← (prompt vide) — bg + ouvre agent view

C · Depuis le shell

  • claude --bg "<prompt>"
  • claude --agent code-reviewer --bg "..."
  • Retourne un ID court : 7c5dcf5d

8 Isolation des écritures fichiers

Toutes les sessions en arrière-plan lisent le code source partagé, mais leurs écritures sont automatiquement redirigées dans un git worktree isolé sous .claude/worktrees/.

9 Que se passe-t-il si 2 sessions modifient le même fichier ?

Claude Code anticipe le problème en bloquant les écritures directes dans le working directory pour toute session en arrière-plan. Dès qu'une session tente d'éditer un fichier, elle est automatiquement déplacée dans un git worktree isolé sous .claude/worktrees/<session-id>/. Chaque session travaille donc sur sa propre copie, sans se voir mutuellement.

Cas où l'isolation NE s'applique PAS

D'après la doc, le blocage des écritures dans le working directory ne s'applique pas dans 3 cas :

⚠ Session déjà dans un worktree

Si tu as lancé claude depuis .claude/worktrees/X, elle écrit dans ce worktree directement (pas de double isolation).

⚠ Pas un repo git

Écritures directes autorisées. Risque réel de race condition si 2 sessions modifient le même fichier (dernière écriture gagne, pas de garde-fou).

⚠ Écritures hors du working dir

Ex : ~/.config/, /tmp/. Directes aussi → même risque de collision.

Workflow recommandé

1. Dispatch session A : "refactor auth"           → worktree A cree auto
2. Dispatch session B : "add logging to auth"     → worktree B cree auto
3. Les 2 tournent en parallele, chacune commit dans sa branche
4. Session A finit  → PR depuis branche A vers main
5. Session B finit  → PR depuis branche B vers main
6. Tu merges PR A   → main avance
7. Tu rebases B sur main → resous conflits eventuels → merges
8. claude rm <id-A> et <id-B>  ⚠ supprime les worktrees

Pièges à connaître

  • ⚠ claude rm détruit le worktree : merger ou pusher la branche avant de supprimer la session, sinon le travail est perdu.
  • ⚠ Worktrees orphelins : si une session se termine sans cleanup propre, .claude/worktrees/ se remplit. Lister avec git worktree list et nettoyer avec git worktree remove <path>.
  • ⚠ Lectures non synchronisées : la session B ne voit pas les modifs de A tant que A n'a pas mergé sur main puis que B n'a pas rebasé. Deux sessions "parallèles" sur le même fichier travaillent en réalité sur la version d'origine chacune.
  • 💡 Forcer l'isolation pour un subagent : ajouter isolation: worktree dans son frontmatter pour qu'il s'isole toujours, quelle que soit la façon dont il est lancé.
