À lire d'abord

La fameuse table des poids de 2023 — « replies count more than likes by a big multiplier » — décrit un système qui n'existe plus sous cette forme. L'algorithme 2026 est un transformeur (Phoenix) qui apprend les poids depuis votre historique d'engagement, scoré contre une surface multi-actions, gatée par un service offline de compréhension de contenu (Grox). La forme du scoring importe désormais beaucoup plus que les nombres, et les nombres eux-mêmes ne sont pas dans le release public. Ce rapport s'en tient à ce que le code montre.

Trois composants font presque tout le travail :

Composant · 01

Home Mixer

x-algorithm/home-mixer/

Orchestrateur Rust qui exécute le pipeline ci-dessus à chaque requête.

Composant · 02

Thunder

x-algorithm/thunder/

Store Rust en mémoire des posts récents par utilisateur, alimenté par Kafka. Lookups sub-millisecondes des candidats in-network (thunder/main.rs, thunder/post_store.rs).

Composant · 03

Phoenix

x-algorithm/phoenix/

Service ML JAX qui fait à la fois la récupération de candidats (modèle two-tower) et le ranking (transformeur dérivé de Grok-1).

Un quatrième composant, Grox (x-algorithm/grox/), fonctionne en offline — il classifie et encode les posts sur un chemin séparé, écrit ces signaux dans le feature store, et Home Mixer les hydrate plus tard. Grox ne se trouve pas sur le chemin chaud des requêtes.

1.2 Les 19 choses que Phoenix prédit (et pourquoi c'est le changement majeur)

L'algorithme de 2023 scorait chaque candidat contre une combinaison linéaire d'étiquettes d'engagement faite à la main, avec des multiplicateurs largement fuités. L'algorithme 2026 score chaque candidat contre la sortie d'un transformeur appris qui prédit les probabilités de dix-neuf actions différentes. D'après phoenix/runners.py:233-252 et confirmé dans home-mixer/candidate_pipeline/candidate.rs:30-51 (struct PhoenixScores) :

Le changement majeur par rapport à 2023 est la surface d'actions elle-même. Nouveaux signaux first-class : dwell_time continu (et plus seulement binaire dwelled / not-dwelled), vqv_score (un visionnage vidéo gated par un seuil de qualité, pas un visionnage brut), follow_author_score (le modèle apprend à prédire si un post fera gagner un nouveau follower à son auteur), et trois variantes distinctes de partage (in-app, via DM, via copy-link). Chacune est une prédiction séparée ; chacune peut être pondérée indépendamment dans la formule de score.

1.3 La formule de score — et le caveat sur les poids

Le Weighted Scorer combine les prédictions exactement comme le README l'écrit : Σ (weight × P(action)). Le câblage complet est dans home-mixer/scorers/weighted_scorer.rs:44-91. Concrètement, le score d'un candidat est :

favorite × FAVORITE_WEIGHT
+ reply × REPLY_WEIGHT
+ retweet × RETWEET_WEIGHT
+ photo_expand × PHOTO_EXPAND_WEIGHT
+ click × CLICK_WEIGHT
+ profile_click × PROFILE_CLICK_WEIGHT
+ vqv × (VQV_WEIGHT if video duration > MIN_VIDEO_DURATION_MS else 0)
+ share × SHARE_WEIGHT
+ share_via_dm × SHARE_VIA_DM_WEIGHT
+ share_via_copy_link × SHARE_VIA_COPY_LINK_WEIGHT
+ dwell × DWELL_WEIGHT
+ quote × QUOTE_WEIGHT
+ quoted_click × QUOTED_CLICK_WEIGHT
+ dwell_time × CONT_DWELL_TIME_WEIGHT
+ follow_author × FOLLOW_AUTHOR_WEIGHT
+ not_interested × NOT_INTERESTED_WEIGHT
+ block_author × BLOCK_AUTHOR_WEIGHT
+ mute_author × MUTE_AUTHOR_WEIGHT
+ report × REPORT_WEIGHT

Les quatre poids négatifs sont des nombres négatifs, donc ils soustraient. Une petite étape offset_score() (weighted_scorer.rs:83-91) maintient les scores agrégés négatifs bien ordonnés par rapport aux positifs.

Le caveat — et c'est le caveat le plus important de ce rapport

Aucune de ces constantes *_WEIGHT n'est définie dans le release open-source. Elles sont toutes référencées comme crate::params::FAVORITE_WEIGHT etc., mais aucun module params.rs n'existe dans le code publié (vérifié par grep -rn sur tout le repo). Les poids sont gérés par un service feature-switch / paramètres que X opère en interne, qui leur permet de faire de l'A/B test et d'ajuster les poids sans changement de code.

Quiconque vous dit « les replies valent N,N× plus que les likes en 2026 » fabrique un chiffre qui n'est pas dérivable du release OSS.

Ce que ce rapport peut vous dire — et qui est réellement utile — c'est le signe et le rôle structurel de chaque poids : quelles actions sont positives, lesquelles sont négatives, lesquelles sont gatées par des propriétés de contenu, et à quelles surfaces elles s'appliquent.

1.4 Les multiplicateurs cachés

Trois multiplicateurs structurels agissent sur le score après la somme pondérée. Ils ne sont pas dans la grille « table de poids » parce qu'ils multiplient l'ensemble du score, pas des probabilités d'action individuelles — mais ils façonnent les résultats plus que n'importe quel poids individuel.

Multiplicateur · 01

Pénalité Out-of-Network (OON)

home-mixer/scorers/oon_scorer.rs:20-23

score = base_score × OON_WEIGHT_FACTOR  si OON
score = base_score                       sinon

Le README indique OON_WEIGHT_FACTOR < 1, et le commentaire de la ligne 7 confirme l'intention : « Prioritize in-network candidates over out-of-network candidates. » C'est la version codifiée de « les followers comptent » : un post d'un compte que vous suivez est noté contre un plafond plus élevé que le même post venant d'un compte que vous ne suivez pas.

Multiplicateur · 02

Exception OON pour les nouveaux utilisateurs

home-mixer/scorers/ranking_scorer.rs:220-239

Pour les utilisateurs dont l'âge du compte est inférieur à NewUserAgeThresholdSecs et qui suivent au moins NEW_USER_MIN_FOLLOWING comptes, la pénalité OON est remplacée par NEW_USER_OON_WEIGHT_FACTOR. Les nouveaux comptes qui ont fait un onboarding minimal (suivre suffisamment de gens) obtiennent un traitement différent du contenu out-of-network. Cela biaise ce que voient les nouveaux utilisateurs — et par extension, ce que les nouveaux comptes peuvent atteindre.

Multiplicateur · 03

Atténuation de diversité d'auteurs

home-mixer/scorers/author_diversity_scorer.rs:29-31

multiplier(position) =
  (1 - floor) × decay_factor^position + floor

Où position est le nombre de posts de cet auteur déjà apparus plus haut dans la liste triée de candidats. Le meilleur post d'un auteur donné dans un render donné garde tout son score ; chaque post suivant du même auteur est atténué, et l'atténuation ne descend jamais sous floor. L'implication pour la cadence de publication est structurelle, pas heuristique.

Multiplicateur · 04

Gate de durée vidéo

home-mixer/scorers/weighted_scorer.rs:72-81

Le vqv_score (video quality view) n'entre dans la somme pondérée que si la durée de la vidéo dépasse MIN_VIDEO_DURATION_MS. Les clips plus courts ne reçoivent simplement pas du tout la contribution VQV (leur poids vqv est remplacé par zéro). C'est un gate dur, pas une pénalité douce.

1.5 Ce qui n'entre jamais dans le ranker

Deux couches de filtres tiennent les posts complètement à l'écart du pipeline de scoring. La reach perdue ici est invisible — le post n'a jamais été scoré, il n'a donc jamais eu la chance de bien scorer.

Filtres pre-scoring

home-mixer/filters/, exécutés séquentiellement avant tout scoring :

  • DropDuplicatesFilter — IDs de posts exactement dupliqués.
  • CoreDataHydrationFilter — posts qui ont échoué à s'hydrater.
  • AgeFilter — plus anciens que l'âge max configuré.
  • SelfpostFilter — vos propres posts.
  • RepostDeduplicationFilter — multiples reposts du même contenu.
  • IneligibleSubscriptionFilter — contenu payant pour les non-abonnés.
  • PreviouslySeenPostsFilter — posts dans les seen_ids du viewer.
  • PreviouslyServedPostsFilter — posts déjà servis dans la session courante.
  • MutedKeywordFilter — posts matchant les muted keywords du viewer.
  • AuthorSocialgraphFilter — posts d'auteurs que le viewer a bloqués ou mutés, et posts où l'auteur bloque le viewer (y compris l'auteur cité d'un quote tweet). La relation de block est vérifiée dans les deux sens.

Filtres post-selection

Après ranking, avant servir :

  • VFFilter — visibility filtering : deleted, spam, violence, gore.
  • DedupConversationFilter — replie les multiples branches d'une même conversation.

Gates offline Grox

x-algorithm/grox/, exécutés séparément et écrits au feature store avant qu'un post ne devienne candidat. Ce sont des classifieurs d'éligibilité, pas des modificateurs de score :

  • task_spam_detection.py / classifiers/content/spam.py — écran spam Grok-based pour les replies à faible follower / faible qualité.
  • task_post_safety_screen_deluxe.py — écran violence, NSFW, misinformation.
  • task_safety_ptos_policy.py — violations des conditions d'utilisation.
  • task_banger_screen.py — écran haute qualité / viral en amont (positif).
  • task_reply_ranking.py — scoring qualité des replies, consommé pour le ranking de conversation.

Le point architectural

Le contenu borderline n'est pas « rétrogradé » dans le système 2026. Il est disqualifié au moment de l'hydratation et n'atteint jamais le scoring. C'est matériellement différent du modèle reduced-reach / visibility-filtering du système 2023.

1.6 Ce que Phoenix apprend réellement

L'input le plus conséquent de Phoenix n'est pas le post candidat ; c'est votre séquence d'actions. home-mixer/query_hydrators/user_action_seq_query_hydrator.rs récupère l'historique d'engagement récent du viewer (likes, replies, shares, dwell, blocks, mutes, etc.) et le passe comme séquence au transformeur en plus de l'ensemble des candidats. D'après phoenix/recsys_model.py et le design de candidate-isolation décrit dans le README : les candidats peuvent attendre votre séquence d'actions, mais ils ne peuvent pas s'attendre les uns les autres.

Les implications :

  • Les scores sont per-user et per-post, pas globaux. Le même post prédit à gagner un like de l'utilisateur A à la probabilité p₁ est prédit à gagner un like de l'utilisateur B à la probabilité p₂ ; il n'y a pas de chiffre unique « ce post est bon ».
  • « Devenir viral » est un résultat au niveau population de nombreuses prédictions personnelles indépendantes qui franchissent le seuil, pas une métrique unique que Phoenix optimise.
  • Récence, affinité de sujet, proximité sociale, préférence linguistique et affinité historique d'auteur sont toutes implicites dans la séquence. Ce ne sont plus des features codées à la main — le README, ligne 55 : « We have eliminated every single hand-engineered feature and most heuristics from the system. » Le transformeur les apprend.
  • Deux signaux associés bénéficient d'une hydratation explicite même dans ce monde transformer-first : mutual_follow_jaccard_hydrator.rs calcule la similarité de Jaccard entre les followers du viewer et les followers de l'auteur candidat (proximité sociale), et following_replied_users_hydrator.rs signale quand un utilisateur que le viewer suit a répondu à un candidat. Ce sont des échafaudages que le transformeur peut utiliser.

Pour la récupération OON (two-tower Phoenix), l'input est aussi la séquence d'actions : votre historique d'actions devient un embedding utilisateur, les posts candidats ont des embeddings post, et le système récupère les posts dont les embeddings sont proches du vôtre. Les embeddings viennent en partie de grox/embedder/multimodal_post_embedder_v5.py, qui encode texte + images + (optionnel) transcription ASR du vidéo en un vecteur normalisé de 1024 dimensions.

Votre découvrabilité par des inconnus est donc fonction de la proximité d'embedding entre vos posts et ceux que votre audience cible engage déjà.

1.7 Ce qui a changé depuis le release 2023

Cinq différences sont suffisamment grandes pour être structurantes dans la réflexion growth :

  1. Pas de feature codée à la main. Le release 2023 avait des termes de boost / pénalité explicites pour le statut verified, les abonnés payants, la distance réseau, etc. Le release 2026 les a arrachés au profit de laisser le transformeur apprendre la pertinence depuis les séquences d'actions (README ligne 55, plus la section Key Design Decisions). Cela signifie que les conseils « règle du pouce » liés à des multiplicateurs spécifiques de 2023 ne sont pas seulement datés — ils pointent vers du code qui n'existe plus.
  2. Un modèle prédisant plusieurs actions, au lieu de plusieurs modèles prédisant chacun une action. Phoenix prédit le vecteur complet à dix-neuf actions en une seule forward pass avec un masking de candidate-isolation (les scores sont donc batch-independent et cacheables). Le système 2023 cousait des prédicteurs séparés ensemble ; Phoenix est unifié.
  3. Grox sépare la compréhension de contenu et le ranking. Spam, safety, PTOS et classification « banger » se passent toutes sur un chemin offline séparé (grox/plan_master.py orchestre neuf plans parallèles) et écrivent dans des feature stores. Home Mixer lit ces signaux comme des gates déjà décidés. Le chemin chaud ne fait pas de compréhension de contenu ; ce travail a été déplacé en amont.
  4. Nouveaux signaux d'engagement first-class. dwell_time continu, video quality view (vqv_score), follow_author_score et trois variantes de share (share, share_via_dm, share_via_copy_link) n'existaient pas comme poids discrets dans le release 2023. Chacun a un poids séparé, et plusieurs concernent des comportements que les créateurs peuvent directement influencer.
  5. Récupération OON two-tower. La découverte out-of-network n'est plus « SimClusters + heuristiques » ; c'est un modèle two-tower appris où à la fois les utilisateurs et les posts ont des embeddings, et la recherche ANN récupère les candidats proches (phoenix_source.rs, phoenix/recsys_retrieval_model.py). L'embedding d'un post vient d'un modèle multimodal qui traite texte, images et optionnellement une bande sonore transcrite de toute vidéo. Cela rend la clarté sémantique d'un post mesurablement plus importante que le simple chevauchement de mots-clés.

Une note sur la confiance

Les affirmations sur la direction (poids positif ou négatif, gate dur ou ajustement doux, existe ou pas) sont à haute confiance et citées. Les affirmations sur la magnitude (combien, à quelle fréquence, dans quel ratio) sont marquées (directional) parce qu'elles ne peuvent pas être dérivées de l'OSS.

2.A · Voix personnelle / fondateur

Vous êtes un seul humain qui poste depuis un seul compte. Votre objectif est une combinaison de croissance de followers, compounding de réseau et profondeur conversationnelle. Votre budget temps est petit. Choisissez les piliers qui font bouger l'aiguille.

01

Avantage in-network

Tant que vous n'avez pas une vraie base de followers, chaque post que vous publiez lutte contre la pénalité OON (oon_scorer.rs:21). La seule action à plus fort levier que vous puissiez faire est de convertir vos relations réelles existantes en follows — DMs, cross-promo podcast, footers de newsletter, replies sur des comptes dont l'audience chevauche la vôtre. Chaque follower vous convertit de « en compétition dans le pool OON » à « servi par Thunder par défaut » pour cet utilisateur.

02

Cadre P(action)

Arrêtez d'optimiser pour les likes totaux ; les likes totaux sont un artefact de population. Optimisez pour que la prochaine personne qui voit ce post soit susceptible de faire l'action que vous voulez qu'elle fasse. Choisissez une seule action par post : un reply (vous posez une question), un follow (vous postez une ligne d'autorité qui leur donne envie d'en voir plus), un dwell (vous postez un thread qui tire l'œil vers le bas), un share (vous packagez un takeaway si proprement que quelqu'un va le coller dans son groupe). Quand vous ne savez pas articuler quelle action vous visez, le post est peu susceptible de bien scorer pour aucune d'elles.

03

Déclencher délibérément les actions à fort poids

Parmi les 19 actions, celles sur lesquelles un compte personnel a le plus d'influence :

  • Reply — les questions ayant une réponse spécifique et défendable récoltent des replies ; les prompts vagues du style « what do you think? » n'en récoltent pas.
  • Dwell time — les threads et posts plus longs récoltent directement du dwell_time ; les posts d'une ligne ne le peuvent pas. Il y a maintenant un signal de dwell continu (index 18), pas seulement binaire dwelled / not-dwelled, donc faire que le post mérite d'être lu lentement compte.
  • Profile click — le post qui donne envie à un inconnu de savoir qui vous êtes. Signalisation alignée sur la bio, ligne d'identité forte, revendication de domaine.
  • Follow-author — Phoenix a sa propre prédiction pour ça (index 13). Les posts qui se lisent comme « voici ma prémisse et je vais la défendre à travers de nombreux posts à venir » récoltent ça ; les observations one-off n'en récoltent pas.
04

Survivre à l'atténuation de diversité

L'author diversity scorer (author_diversity_scorer.rs:29-31) trie les candidats par score puis atténue chaque apparition répétée du même auteur dans ce render unique. Règles pratiques :

  • Le meilleur post unique que vous publiez dans une fenêtre donnée garde tout son score. Le second-meilleur est atténué. Donc publiez votre meilleur post en premier, pas en dernier.
  • N'empilez pas les posts dos à dos ; étalez-les sur les fenêtres où votre audience rafraîchit son fil. (Les paramètres réels de décroissance sont externes — mais le fait structurel que la cadence compte est dans le code.)
  • Les reposts et quotes d'autres auteurs ne déclenchent pas votre décroissance d'auteur ; ils contribuent au score de diversité sous l'ID de l'auteur original.
05

Rester hors de la zone des signaux négatifs

Quatre poids négatifs directs (not_interested, block_author, mute_author, report, indices 14-17). Un mute ou un block soustrait du score de votre contenu futur pour cet utilisateur (parce que le transformeur a maintenant vu l'action négative dans sa séquence). Tactiquement : évitez la voix contrarian-pour-le-plaisir-de-l'être qui récolte des mutes de gens qui sont d'accord avec vous la plupart du temps. Choisissez vos combats ; n'en fabriquez pas.

06

Récupérabilité two-tower (sidebar)

Si vous voulez atteindre des inconnus, postez sur des sujets où l'historique d'engagement récent de quelqu'un inclut des posts comme les vôtres. L'unicité stylistique compte moins que la proximité topique d'embedding — l'embedder multimodal v5 (grox/embedder/multimodal_post_embedder_v5.py) encode le contenu sémantique, pas votre voix de prose. Cross-poster la même idée en trois variantes (texte, image, vidéo) crée trois tirs d'embedding pour être récupéré.

07

Gates Grox / VF (sidebar)

Le contenu borderline ne sous-performe pas ; il n'entre pas du tout dans le ranker. Si un post est à la limite de la classification spam (trop de @mentions, patterns de link spam, replies à faible engagement répétées sur de gros comptes), il peut disparaître entièrement du pool de candidats pour certains utilisateurs. Citez grox/classifiers/content/spam.py si un créateur pousse en retour.

Checklist 5 items pour vos 10 prochains posts

  1. Chaque post nomme l'action unique qu'il vise (reply / dwell / follow / share).
  2. Au moins trois sont des threads ou du long-form (qui récoltent directement du dwell_time).
  3. Aucun n'est posté dans l'heure suivant un autre de vos posts (protection contre l'atténuation de diversité).
  4. Deux sont cadrés pour récolter des profile clicks (identity bait, revendication de domaine).
  5. Zéro sont des prises de position négatives réactives qui risquent de récolter des mutes de votre audience chaude.

2.B · Compte marque / entreprise

Vous représentez une organisation. Les décisions passent par approbation. Vous livrez un calendrier éditorial. Votre objectif est un mix de reach, click-through vers une propriété que vous contrôlez et génération de leads. Vous avez moins de flexibilité de voix qu'un compte personnel mais plus de capacité de production et des KPIs plus clairs.

01

Avantage in-network

Même pénalité (oon_scorer.rs:21), outils différents. Deux moves pratiques :

  • Amplification employés — chaque employé qui poste pour le compte de la marque est une source Thunder additionnelle pour ses followers. La reach du compte marque est l'union de la distribution in-network de chaque employé.
  • Densité de mutual-follow — mutual_follow_jaccard_hydrator.rs calcule la similarité de Jaccard entre les followers du viewer et les followers de l'auteur candidat. Les marques implantées dans les bonnes communautés (vos clients se suivent les uns les autres) bénéficient d'un boost de ce signal qu'un compte marque isolé n'a pas.
02

Cadre P(action)

Arrêtez de remonter les totaux d'impressions au management. Reportez la distribution d'actions par post : quelle action ce post a-t-il récoltée le plus, et était-ce l'action que vous visiez ? Si vous vouliez des clicks mais avez eu des likes, le post était hors-brief — les likes ne sont pas un KPI de marque et Phoenix sous-classera le post pour vos audiences futures en conséquence.

03

Déclencher délibérément les actions à fort poids

Actions pertinentes pour une marque à concevoir :

  • Click (index 4) et quoted click (index 12) — les actions qui drivent du trafic site mesurable. Les posts qui promettent un payoff nécessitant de quitter X récoltent ça.
  • VQV (index 6) — vues vidéo au-delà de MIN_VIDEO_DURATION_MS. Couper une vidéo trop court tue la contribution entièrement. Construisez des assets vidéo assez longs pour franchir le seuil et montés assez serrés pour tenir l'attention au-delà. Hook court, puis substance.
  • Share via DM (index 8) et share via copy-link (index 9) — ce sont des actions distinctes des shares in-app (index 7). Les posts qui sont collés dans des conversations privées et des DMs Slack ont assez de valeur pour mériter leurs propres poids. Décortications de pricing, comparaisons de produits, démos « regarde ça », controverse que les gens forwardent en privé plutôt que de répondre publiquement.
  • Follow-author (index 13) — concevez au moins un post par semaine comme un follow magnet : « On est l'équipe qui construit X. Ce thread est une visite de ce qu'on shippe le mois prochain. » Phoenix prédit follow-author comme une probabilité discrète et le pondère en conséquence.
04

Survivre à l'atténuation de diversité

Les marques violent souvent ça sans s'en rendre compte :

  • Un « drumbeat » de quatre posts espacés de 30 minutes sous-performe le meilleur d'entre eux, seul (author_diversity_scorer.rs:29-31).
  • Le compte marque et le CEO postant la même annonce comptent comme deux auteurs distincts — ça bat un seul compte postant deux fois. Coordonnez à travers les humains, n'empilez pas sur un seul handle.
  • Pour les lancements, postez l'annonce une fois ; laissez les employés et l'amplification faire la couverture multi-sources.
05

Rester hors de la zone des signaux négatifs

Poids négatifs directs (indices 14-17) plus les gates plus durs : AuthorSocialgraphFilter inclut le cas « l'auteur bloque le viewer » pour les retweets et quotes, donc interagir avec des comptes controversés qui bloquent votre audience peut supprimer votre reach dans des directions inattendues. Pour la brand safety spécifiquement : le release du 15 mai 2026 a ajouté un module home-mixer/ads/ avec un tracking brand-safety qui respecte les frontières de contenu sensible — ce qui signifie que si vos posts organiques atterrissent à répétition près de sujets sensibles, vous risquez d'être adjacents à des filtres de brand-safety qui affectent à la fois votre reach et votre monétisation.

06

Récupérabilité two-tower

C'est le levier le plus sous-utilisé pour les marques. Tactiques concrètes :

  • Cohérence topique sur les semaines. L'embedding de votre compte, en agrégé, est ce qui détermine à quelles queries de récupération vos posts apparaissent. Poster sur trop de sujets dilue l'embedding.
  • Posts multimodaux. multimodal_post_embedder_v5.py encode texte + image + (optionnel) transcription ASR vidéo. Un post avec une image forte et une caption claire obtient un embedding plus riche que du texte seul. Un post avec une vidéo de quelqu'un parlant clairement les mots-clés obtient la contribution ASR. Sous-titrez vos vidéos pour le modèle, pas seulement pour le viewer.
  • Alignement topic-pack. Les nouvelles sources de candidats incluent phoenix_topics_source.rs et phoenix_moe_source.rs — être identifiable à un topic aide vos posts à apparaître via ces sources, pas seulement via la récupération générique.
07

Gates Grox / VF

Pour les comptes marque c'est de la compliance, pas de la créativité :

  • task_post_safety_screen_deluxe.py et task_safety_ptos_policy.py ne sont pas ajustables. Votre post legal-cleared peut quand même se faire exclure silencieusement s'il déclenche le classifieur. Mettez en place une watchlist de posts qui sous-performent mystérieusement et auditez-les comme exclusions suspectées, pas comme mauvais posts.
  • task_spam_detection.py keye sur des patterns de replies à faible follower / faible qualité. Les comptes marque dont la stratégie d'engagement implique beaucoup de replies sur de gros posts peuvent accidentellement se faire flagger. Répondez substantiellement (le modèle évalue le contexte) ou pas du tout.

Checklist 5 items pour vos 10 prochains posts

  1. L'intention d'action est annotée dans le calendrier (target_action: click | vqv | dm_share | follow | reply).
  2. Chaque vidéo franchit MIN_VIDEO_DURATION_MS et est sous-titrée (l'embedder bénéficie de la transcription ASR-readable).
  3. Un plan de distribution cross-author existe pour la semaine (marque + au moins 2 employés, jamais empilés).
  4. Les posts ont une thématique topique sur la semaine, pas du saut aléatoire de sujet (cohérence d'embedding).
  5. Au moins un post est conçu spécifiquement comme un follow magnet pour l'audience du lancement du mois prochain.

2.C · Cadre généralisé

Pour les audiences qui veulent le modèle indépendamment du rôle. Cette section est l'abstraction durable ; 2.A et 2.B sont les tactiques spécifiques au rôle.

Les trois couches de reach

L'algorithme 2026 est une pile de trois gates indépendants. Un post doit passer les trois pour atteindre un utilisateur donné :

Couche · 01

Éligibilité

Filtres pre-scoring et classifieurs offline Grox (Partie 1.5). Binaire : dedans ou dehors. Aucune optimisation possible ici hors « ne pas se faire exclure ».

Couche · 02

Récupération

Pour l'OON, le modèle two-tower (Partie 1.6). Probabiliste : votre post est ou n'est pas près de l'embedding de l'utilisateur. Optimisé par ce que vous postez (sujet, contenu sémantique, complétude multimodale).

Couche · 03

Ranking

La somme pondérée de dix-neuf probabilités d'actions apprises, modifiée par les multiplicateurs OON / diversité / video-gate (Parties 1.3, 1.4). Continu : votre post score relativement aux autres candidats. Optimisé par comment vous postez (quelle action vous déclenchez, quand, à quelle cadence).

La plupart des conseils growth confondent la couche 3 avec tout le système. C'est la couche la plus visible mais c'est la dernière couche ; les couches 1 et 2 déterminent si vous êtes même arrivé à être scoré.

La surface d'actions comme question produit

Phoenix prédit dix-neuf actions parce que les utilisateurs font dix-neuf choses distinctes. Chaque action est une relation différente que l'utilisateur peut avoir avec un post : je vais faire ça socialement (like, reply, quote, share), je vais faire ça en privé (DM-share, copy-link-share, dwell, photo expand), je déclare une identité (follow author, profile click), je rejette (not-interested, block, mute, report). Un post qui ne choisit pas une relation cible est prédit près de zéro pour toutes et score en conséquence.

Le transformeur comme mécanisme d'équité

Parce que Phoenix prédit des probabilités per-user depuis des séquences d'actions per-user, le « bon contenu » n'est pas une propriété globale. Deux posts qui semblent identiques à un humain peuvent avoir des résultats de ranking radicalement différents pour le même utilisateur à cause de ce que cet utilisateur a historiquement engagé. C'est mécaniquement équitable (pas de biais global vers les comptes établis au-delà du facteur OON) et personnellement inscrutable (aucun créateur ne peut savoir exactement pourquoi un post a marché et pas un autre). La formulation honnête est ciblage probabiliste, pas qualité déterministe.

L'atténuation de diversité d'auteurs comme feature structurelle

C'est la réponse de l'algorithme à la monoculture : empêcher qu'un seul créateur ne domine le fil d'un utilisateur dans un seul render. L'implication pour les créateurs est contre-intuitive :

Publier plus de posts par jour au-delà d'un seuil réduit, n'augmente pas, la reach totale — chaque post marginal est atténué et entre en compétition avec vos posts précédents dans le même render.

La cadence optimale est donc non-monotone en nombre de posts ; il y a un plateau par jour fixé par le taux de rafraîchissement de l'audience, la position de l'auteur, et les paramètres de (1 - floor) × decay^position + floor.

Les signaux négatifs comme veto one-shot

Un mute ou un block ne coûte pas seulement le post qui l'a déclenché ; il apparaît comme une entrée négative dans la séquence d'actions de cet utilisateur pour toujours, ce que Phoenix utilise pour chaque décision de scoring future concernant votre contenu pour cet utilisateur. Le coût est donc cumulatif sur tous vos posts futurs pour cet utilisateur. Les mutes évitables d'audiences sympathiques sont extrêmement coûteux en valeur espérée.

L'exclusion au moment de l'éligibilité est le tueur silencieux

Le contenu borderline en 2026 ne sous-performe pas ; il disparaît. Il n'y a aucun signal au créateur qu'un classifieur Grox a exclu son post des pools de candidats. Le seul diagnostic est l'analyse comparative de reach à travers des posts à distance variable des bords de la policy.

Deux lois de la croissance mécanique

Loi · 01

L'in-network est multiplicatif, l'OON est additif

Les followers transforment chaque post futur d'un évènement probabiliste de récupération OON en un lookup Thunder déterministe. Les retours sur l'acquisition de followers compoundent sur chaque post que vous publierez jamais.

Loi · 02

Le job du modèle est de vous prédire, pas de vous récompenser

Phoenix optimise pour la précision prédictive de l'action utilisateur, pas pour le succès créateur. Les stratégies de croissance qui combattent ce que les utilisateurs font prévisiblement (mendier de l'engagement, manipuler avec de la controverse) blessent la qualité de prédiction de votre compte aux yeux du modèle et déclassent vos posts futurs en conséquence.

2.D · Livrable client / consulting

Pour usage externe. Chaque affirmation cite le fichier où vit la preuve. La formulation est « ce que nous observons dans le release source public, et ce que cela implique pour des interventions de croissance mesurables ».

Résumé exécutif

L'algorithme For You X 2026 est un système à trois étages : un service offline de compréhension de contenu (Grox) gate l'éligibilité, un modèle de récupération two-tower fait remonter les candidats out-of-network, et un transformeur dérivé de Grok-1 (Phoenix) classe les candidats en prédisant dix-neuf actions d'engagement distinctes par utilisateur. Le ranking est Σ (weight × P(action)) modifié par une pénalité out-of-network multiplicative et une atténuation de diversité d'auteurs (citations ci-dessous). Les poids numériques ne sont pas dans le release ouvert. Les interventions de croissance devraient cibler les leviers structurels documentés dans le code : quelle action un post est conçu pour déclencher, si le contenu passe les classifieurs d'éligibilité, si l'embedding du post se trouve dans le bon voisinage pour l'audience cible, et si la cadence de publication respecte l'atténuation de diversité.

Interventions recommandées (ordonnées par impact attendu, toutes directionnelles en attente de mesure client)

  1. Conception de contenu ciblée par action. Réorganiser le calendrier éditorial du client de sorte que chaque post déclare une action cible parmi les dix-neuf. Mesurer le succès par distribution d'actions par post, pas par impressions brutes. Effet attendu : amélioration des scores prédictifs au niveau post pour le segment utilisateur cible parce que le contenu est congruent avec l'action contre laquelle il est scoré.
  2. Playbook de distribution d'auteurs. Auditer la cadence de publication du client par rapport à l'atténuation de diversité d'auteurs. Remplacer les drumbeats en mono-compte empilés par une amplification multi-compte distribuée à travers les employés, partenaires et le handle marque. Effet attendu : chaque post individuel entre dans le ranker à position = 0 pour son auteur dans un render donné plutôt qu'à position = 1, 2, ....
  3. Spécification vidéo. Établir une longueur de cut minimale au-dessus de MIN_VIDEO_DURATION_MS (la valeur exacte est externe — le client doit faire de l'A/B test de longueurs de cut pour identifier le seuil empiriquement pour son contenu). Toutes les vidéos shippées avec audio ASR-friendly et sous-titres incrustés pour alimenter à la fois vqv et le chemin transcription de l'embedder multimodal.
  4. Cohérence d'embedding topique. Auditer les 90 derniers jours du client pour la largeur topique. Recommander un split 70/30 entre un cœur topique cohérent (qui construit la densité de voisinage d'embedding avec l'audience cible) et du contenu exploratoire (qui fait remonter de nouvelles audiences mais dilue l'embedding).
  5. Audit du gate d'éligibilité. Échantillonner des posts de l'archive du client qui ont sous-performé par rapport à leur cohorte et les vérifier contre les types de gate publiquement documentés (patterns de spam, frontières de safety, surface PTOS). Traiter les exclusions silencieuses suspectées comme un mode de défaillance séparé de « mal classé ».
  6. Hygiène des signaux négatifs. Identifier les patterns antagonistes récurrents du client et quantifier leur coût attendu comme cumulatif sur les posts futurs vers l'audience mutée. Recommander de retirer les patterns dont le taux de mute de l'audience chaude dépasse un seuil interne.
  7. Recadrage de l'acquisition de followers. Recadrer la dépense d'acquisition de followers comme l'achat d'une reach future multiplicative (via la conversion OON-vers-Thunder) plutôt que comme l'achat d'une métrique d'audience unique. Le facteur OON est une taxe permanente jusqu'à ce que l'utilisateur suive ; un follow soulève chaque post futur pour cet utilisateur.

Limites et déclarations d'honnêteté (obligatoires pour le livrable)

Les poids numériques ne sont pas dérivables du release OSS. Toutes les affirmations « X est plus important que Y » dans ce rapport décrivent la structure du code (signe, gate, présence d'un multiplicateur), pas la magnitude. Les artefacts modèle publiés sont un mini-checkpoint Phoenix (2 couches, 4 têtes d'attention, 256-dim selon le changelog du 15 mai 2026 et phoenix/README.md), pas les poids de production, donc l'inférence directe contre le code publié démontre la forme du comportement plutôt que les scores de production. Plusieurs intégrations sont stubbées : des marqueurs panic!("Not implemented: to_thrift for ...") existent dans home-mixer/candidate_pipeline/candidate_features.rs et user_features.rs, et les listes de marques brand-safety, mappings d'IDs de topics, pénalités linguistiques et règles de blending ads ne sont pas présents dans le release public. Le client doit traiter ce rapport comme un modèle structurel, pas comme un prédicteur quantitatif.

A.2 · Glossaire

Phoenix

Le transformeur dérivé de Grok-1 qui fait à la fois le ranking et la récupération. Prédit 19 actions par candidat.

Thunder

Le service Rust en mémoire qui stocke les posts récents et sert les candidats in-network avec une latence sub-milliseconde.

Grox

Le service offline de compréhension de contenu. Exécute des classifieurs (spam, safety, PTOS) et des embedders (multimodal v5). Écrit les résultats au feature store pour que Home Mixer les lise.

Home Mixer

L'orchestrateur Rust qui exécute le pipeline au moment de la requête : hydrate → source → hydrate → filter → score → select → filter.

OON

Out-of-network. Un candidat venant d'un compte que le viewer ne suit pas. Soumis à une pénalité de score multiplicative.

VQV

Video quality view. Une vue vidéo qui franchit la durée minimale configurée. Le poids associé contribue uniquement quand video_duration_ms > MIN_VIDEO_DURATION_MS.

Two-tower

Un pattern de récupération ML où utilisateurs et items sont encodés par des tours séparées dans un espace d'embedding partagé, puis matchés par recherche du plus proche voisin.

Candidate isolation

Le pattern de masking d'attention dans le ranking Phoenix où les candidats peuvent attendre le contexte de l'utilisateur mais pas l'un l'autre, rendant les scores batch-independent et cacheables.

VFFilter

Le filtre de visibilité post-selection qui supprime le contenu deleted, spam, violence et gore.

A.3 · Frontière d'honnêteté (ce qu'on ne peut pas savoir depuis l'OSS)

Ce rapport évite délibérément plusieurs affirmations qui nécessiteraient des informations absentes du release public :

  • Valeurs numériques des poids. FAVORITE_WEIGHT, REPLY_WEIGHT, RETWEET_WEIGHT, OON_WEIGHT_FACTOR, NEW_USER_OON_WEIGHT_FACTOR, AUTHOR_DIVERSITY_DECAY, AUTHOR_DIVERSITY_FLOOR, MIN_VIDEO_DURATION_MS, NEGATIVE_SCORES_OFFSET, WEIGHTS_SUM, NEGATIVE_WEIGHTS_SUM, et toute autre constante référencée comme crate::params::* sont de la configuration externe. Le repo ne contient aucun params.rs (vérifié). Tout rapport — y compris les versions futures de celui-ci — qui cite des multiplicateurs numériques spécifiques pour l'algorithme 2026 fabrique des données.
  • Comportement du modèle de production. Le checkpoint Phoenix publié est un mini-modèle (2 couches, 4 têtes, 256 dimensions selon phoenix/README.md) entraîné sur un corpus de 537 000 posts sportifs. Le Phoenix de production est plus gros et continuellement réentraîné sur le trafic d'engagement complet. L'inférence contre les artefacts publiés démontre quels types de features le modèle utilise et comment les scores se comportent structurellement, pas ce que le modèle de production renvoie.
  • Intégrations stubbées. Plusieurs frontières de sérialisation Thrift panic avec "Not implemented: to_thrift for ..." (dans candidate_features.rs et user_features.rs). Le code public ne tourne pas end-to-end contre les services internes de X tel qu'expédié.
  • Données de policy. Listes de marques brand-safety, mappings d'IDs de topics, tables de pénalités linguistiques, règles de blending d'ads, contenus des corpus de muted-keywords, et poids des classifieurs entraînés spam / safety / PTOS ne sont pas dans le release ouvert. Les chemins de code qui les consomment existent ; les données pas.
  • Cadence de réentraînement continu. Le release OSS est un snapshot statique. Le modèle de production est continuellement mis à jour. Toute affirmation sur « ce que fait l'algorithme en ce moment » basée sur la lecture de ce code est au mieux une affirmation structurelle ; les prédictions spécifiques du modèle à tout moment dépendent des mises à jour d'entraînement qui ne sont pas publiques.
